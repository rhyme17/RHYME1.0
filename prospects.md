## 1. 产品概述

### 1.1 产品简介

RHYME 是一款基于 Python 和 PyQt5 开发的纯本地音乐播放器，专注于为用户提供快速、流畅、隐私安全的音乐播放体验。所有功能都在本地实现，无需后端服务，响应速度快，用户体验流畅。

### 1.2 核心价值主张

- **纯本地运行**：无需联网，无需账号，保护用户隐私
- **极致性能**：扫描缓存优化、时长缓存、异步处理，确保流畅体验
- **智能歌词**：三层歌词获取方案（本地LRC → 在线抓取 → 离线ASR识别）
- **模块化架构**：清晰的代码结构，易于维护和扩展
- **跨平台潜力**：当前以 Windows 为主，预留 Linux 和移动端扩展

### 1.3 产品形态

- **当前形态**：Windows 桌面应用程序（.exe 可执行文件）
- **未来规划**：Linux 桌面端、移动端（Android/iOS）、Web 端

## 4. 核心功能与特色

### 4.1 核心功能模块

#### 4.1.1 音乐播放模块

**功能描述**：
- 支持多种音频格式：MP3、FLAC、WAV、AAC、OGG、M4A
- 完整的播放控制：播放、暂停、上一首、下一首
- 音量控制：滑块调节、静音切换、音量记忆
- 播放模式：顺序播放、随机播放、单曲循环
- 进度控制：拖动进度条、快进快退（5秒）
- 播放记忆：重启后恢复上次播放的歌曲和进度

**技术实现**：
- 基于 `pydub` 进行音频解码
- 使用 `sounddevice` 进行音频输出
- 支持高质量输出模式
- 音量统一强度调节

**用户价值**：
- 流畅的播放体验
- 灵活的控制方式
- 无需重复设置

#### 4.1.2 音乐库管理模块

**功能描述**：
- 本地音乐文件扫描（支持递归扫描子目录）
- 按艺术家和专辑分类
- 关键词实时搜索
- 扫描进度实时显示
- 支持取消扫描

**技术实现**：
- 使用 `mutagen` 解析音频元数据
- 扫描缓存优化（基于文件签名）
- 异步扫描（不阻塞UI）
- 元数据缓存机制

**用户价值**：
- 快速构建音乐库
- 便捷的音乐分类浏览
- 高效的搜索体验

#### 4.1.3 歌单管理模块

**功能描述**：
- 多歌单支持
- 歌单增删改查
- 一键生成歌单（基于扫描目录）
- 歌单内歌曲排序（支持拖拽）
- 歌单持久化存储

**技术实现**：
- JSON 格式存储歌单数据
- 内存中的播放列表管理
- 歌单切换与恢复

**用户价值**：
- 灵活的音乐组织方式
- 快速创建个性化歌单
- 歌单数据持久保存

#### 4.1.4 歌词显示模块（三层方案）

**方案A：本地LRC歌词**
- 自动匹配同名 `.lrc` 文件
- 支持同目录 `lyrics/` 子目录
- 支持多种命名规则（标题-艺术家、艺术家-标题）
- 兼容 UTF-8 和 GB18030/GBK 编码

**方案B：在线歌词抓取**
- 自动尝试在线抓取歌词（Gequbao源）
- 成功后保存为本地 `.lrc` 文件
- 失败时保留调试信息

**方案C：离线ASR识别**
- 基于 `faster-whisper` 的离线语音识别
- 支持中文歌词识别
- 繁简规范化（可配置）
- 识别结果缓存为 `.lrc` 文件

**用户价值**：
- 多层次歌词获取保障
- 无需手动下载歌词
- 支持无网络环境

#### 4.1.5 用户体验模块

**界面设计**：
- 深色主题（主窗口、设置页、弹窗）
- 响应式窗口布局
- 可调整的分割器
- 标签页切换（全部音乐、艺术家、专辑）

**交互设计**：
- 键盘快捷键支持
  - 空格：播放/暂停
  - 上/下方向键：音量调节
  - 左/右方向键：快退/快进 5 秒
  - Ctrl+左/右方向键：上一首/下一首
  - Delete：移除选中歌曲
- 系统托盘支持
  - 关闭时隐藏到托盘（可配置）
  - 首次关闭提示选择行为

**个性化设置**：
- 字体风格切换（细体/常规/中粗）
- 歌词字号调节
- 歌词保存目录自定义
- 扫描目录记忆

### 4.2 核心特色

#### 特色一：三层歌词获取方案

RHYME 提供业界首创的三层歌词获取方案，确保用户在任何情况下都能获得歌词：

1. **优先本地**：自动匹配本地 `.lrc` 文件，零延迟
2. **在线补充**：本地无歌词时自动尝试在线抓取
3. **离线识别**：在线失败时可手动触发 ASR 识别

**竞争优势**：
- 相比主流播放器，提供更全面的歌词获取保障
- 支持无网络环境下的歌词生成
- 用户可自主选择歌词来源

#### 特色二：极致性能优化

**扫描缓存优化**：
- 基于文件签名（大小+修改时间）的缓存机制
- 重复扫描同一目录时复用未变化文件的元数据
- 显著降低二次扫描耗时

**时长读取缓存**：
- 优先使用元数据时长并缓存结果
- 减少重复完整解码导致的列表卡顿

**异步处理**：
- 大目录扫描不阻塞界面
- 歌词识别在后台线程执行

**竞争优势**：
- 相比同类播放器，扫描速度更快
- 列表加载更流畅
- 用户体验更佳

#### 特色三：纯本地架构

**隐私保护**：
- 无需账号登录
- 无数据上传
- 无广告干扰

**简化部署**：
- 无需后端服务
- 单个应用即可运行
- 绿色便携（支持自定义数据目录）

**竞争优势**：
- 相比在线音乐平台，保护用户隐私
- 相比需要登录的播放器，使用更便捷
- 相比商业软件，无广告干扰

#### 特色四：模块化架构

**代码结构**：
- 按职责拆分的 Mixins 模块
- 编排服务（Orchestration Service）
- 后台工作线程（Worker）
- 清晰的层次结构

**技术优势**：
- 易于维护和扩展
- 便于单元测试
- 支持跨平台复用

**竞争优势**：
- 相比单体架构项目，更易迭代
- 便于社区贡献
- 为跨平台扩展奠定基础

#### 5.2.2 模块化设计（Windows桌面端）

**Mixins 模块**（按职责拆分）：
- `UiMixin`：界面构建与控件信号连接
- `PlaybackMixin`：播放控制、进度条、音量与播放状态机
- `PlaylistMixin`：歌单增删改、切换、拖拽排序、撤销
- `LibraryMixin`：扫描、搜索、分类与库列表渲染
- `LyricsMixin`：歌词加载、识别与刷新
- `PersistenceMixin`：`playlists.json` 与 `settings.json` 读写
- `SettingsMixin`：设置页与设置应用逻辑
- `ShortcutMixin`：键盘快捷键管理
- `TrayMixin`：系统托盘功能
- `LifecycleMixin`：应用生命周期管理

**编排服务**（Orchestration Service）：
- `PlayerInitOrchestrationService`：播放器初始化编排
- `PlayerOrchestrationService`：播放器运行时编排
- `PlaybackOrchestrationService`：播放控制编排
- `LibraryOrchestrationService`：音乐库管理编排
- `ScanOrchestrationService`：扫描任务编排
- `UIBindingOrchestrationService`：UI绑定编排

**后台工作线程**（Worker）：
- `ScanWorker`：音乐文件扫描
- `DecodeWorker`：音频解码
- `LyricsWorker`：歌词识别

### 5.3 数据流设计

#### 5.3.1 音乐扫描流程

```
用户点击"添加音乐"
    ↓
选择音乐目录
    ↓
启动 ScanWorker（后台线程）
    ↓
遍历目录，收集音频文件
    ↓
逐个解析元数据（使用缓存优化）
    ↓
实时报告进度到UI
    ↓
扫描完成，更新音乐库
    ↓
用户选择保存为歌单
```

#### 5.3.2 歌词获取流程

```
播放歌曲
    ↓
尝试加载本地LRC歌词 ──成功──→ 显示歌词
    ↓ 失败
尝试在线抓取歌词 ──成功──→ 保存为本地LRC → 显示歌词
    ↓ 失败
提示用户可手动识别
    ↓ 用户点击"识别歌词"
启动 ASR 识别（后台线程）
    ↓
识别完成，保存为本地LRC
    ↓
显示歌词
```

### 5.4 性能优化策略

#### 5.4.1 扫描优化

**文件签名缓存**：
- 基于文件大小和修改时间生成签名
- 缓存已解析的元数据
- 重复扫描时跳过未变化的文件

**效果**：
- 首次扫描：正常速度
- 二次扫描：速度提升 50%-80%

#### 5.4.2 时长读取优化

**元数据优先**：
- 优先从音频元数据读取时长
- 缓存时长信息
- 减少完整解码次数

**效果**：
- 列表加载更流畅
- 减少内存占用

#### 5.4.3 异步处理

**后台线程**：
- 扫描任务在后台线程执行
- 歌词识别在后台线程执行
- UI 线程保持响应

**效果**：
- 大目录扫描不阻塞界面
- 歌词识别不影响播放

## 9. 技术挑战与应对

### 9.1 性能挑战

#### 挑战一：大型音乐库性能

**问题描述**：
- 大量歌曲时列表渲染卡顿
- 内存占用过高
- 启动速度慢

**应对方案**：
- 实现虚拟滚动，只渲染可见区域
- 懒加载歌曲信息
- 使用 SQLite 替代 JSON 存储大型音乐库
- 后台预加载下一首歌曲
- 及时释放不用的资源

#### 挑战二：音频处理性能

**问题描述**：
- 音频解码耗时
- 可视化效果可能影响播放流畅度
- ASR 识别速度慢

**应对方案**：
- 使用缓存机制
- 后台线程处理
- 优化算法
- 提供可配置的性能选项

### 9.2 功能实现挑战

#### 挑战一：音频可视化

**技术难点**：
- 实时音频频谱分析
- PyQt5 绘图性能
- 与播放同步

**实现方案**：
- 使用 pyaudio 或 sounddevice 获取音频数据
- numpy 进行 FFT 变换
- QPainter 绘制频谱柱状图
- 定时器刷新（60fps）

#### 挑战二：桌面歌词

**技术难点**：
- 窗口置顶和透明
- 不影响其他窗口操作
- 性能优化

**实现方案**：
- 使用 Qt.WindowStaysOnTopHint 标志
- 设置窗口透明度
- 无边框窗口
- 定时刷新歌词

#### 挑战三：歌词时间轴微调

**技术难点**：
- 精确的时间控制
- 实时预览效果
- 保存修改

**实现方案**：
- 提供全局偏移量设置
- 实时预览调整效果
- 保存到 LRC 文件或缓存

### 9.3 数据管理挑战

#### 挑战一：数据迁移

**问题描述**：
- 从 JSON 迁移到 SQLite
- 数据格式兼容性
- 数据备份与恢复

**应对方案**：
- 提供迁移工具
- 保持向后兼容
- 自动备份机制

#### 挑战二：数据一致性

**问题描述**：
- 异常退出导致数据丢失
- 并发访问冲突

**应对方案**：
- 定期自动保存
- 使用事务保证原子性
- 数据完整性检查

### 9.4 开发挑战

#### 挑战一：时间精力有限

**问题描述**：
- 个人开发时间有限
- 难以持续投入

**应对方案**：
- 合理规划开发优先级
- 分阶段实现功能
- 保持代码简洁，便于维护

#### 挑战二：技术学习成本

**问题描述**：
- 新功能需要学习新技术
- 调试和优化耗时

**应对方案**：
- 选择成熟的技术方案
- 参考开源项目实现
- 逐步积累经验
